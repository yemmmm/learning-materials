一、什么是 SSO（单点登录）

一句话定义

SSO（Single Sign-On） = 用户只需登录一次， 就能访问多个相互信任的系统，不需要在每个系统都输入用户名密码。

登录一次 Google 账号 = 自动登录 Gmail / YouTube / Drive。这就是 SSO。

核心角色（三个参与者）

👤 User（用户 / 主体）

想用一个身份访问多个系统的人。只需要记住一套账号密码。

🏢 IdP（身份提供商）

Identity Provider——负责认证用户身份的"权威机构"。存储账号密码，验证你是谁。

🌐 SP / RP（服务提供商）

Service Provider / Relying Party——实际提供服务的应用。它信任 IdP 的认证结果，不再自己验证密码。

二、OIDC 是什么

一句话定义

OIDC（OpenID Connect） = 在 OAuth 2.0 授权协议之上， 加了一层身份认证层。它让客户端能验证用户是谁，并获取用户的基本信息。

OAuth 2.0 vs OIDC——最容易混淆的两个概念

OAuth 2.0OIDC (OpenID Connect)
解决的问题 授权：允许 A 访问 B 的资源 认证：证明用户是谁
类比 🏨 酒店门卡——给你进入房间的权限 🆔 身份证——证明你的身份
核心产出 Access Token（访问令牌） ID Token（身份令牌）+ Access Token
用户信息 不包含用户信息 ID Token 包含sub（用户唯一标识）、姓名、邮箱等
关系 OIDC 是 OAuth 2.0 的上层协议（扩展）

OIDC 中的核心概念

ID Token（JWT）

ID Token 是一个 JWT（JSON Web Token），由 IdP 签发，包含用户身份信息。它是 OIDC 区别于 OAuth 2.0 的关键。

{
  "iss": "https://your-idp.com",    // 签发者（谁签发的）
  "sub": "user_123456",            // 用户唯一标识
  "aud": "system_b_client_id",     // 受众（这个 Token 是给谁用的）
  "exp": 1714137600,                // 过期时间
  "iat": 1714051200,                // 签发时间
  "email": "zhangsan@example.com", // 邮箱（可选）
  "name":  "张三"                   // 姓名（可选）
}
sub（Subject）

用户的唯一标识。你的系统用这个字段把 IdP 用户映射到本地账号。

iss（Issuer）

Token 的签发者 URL。验证 Token 时要确认 iss 是可信的。

aud（Audience）

Token 的目标接收方（client_id）。防止 Token 被用于错误的系统。

exp / iat

过期时间和签发时间。过期 Token 必须被拒绝。

四、OIDC 的核心流程（Authorization Code Flow + PKCE）

这是目前最推荐的流程，适合所有有后端的 Web 应用。

各步骤详解

  1. 用户点击"用 XX 登录"
    用户在你的系统中点击登录按钮，后端生成一个授权URL。
  2. 重定向到 IdP 登录页
    你的系统后端构造 Authorization Request，包含 client_id、redirect_uri、scope（openid profile email）、state（防CSRF）、code_challenge（PKCE）。然后 302 重定向浏览器到 IdP。 
    GET /authorize?
  response_type=code
  &client_id=your_system_client_id
  &redirect_uri=https://your-system.com/callback
  &scope=openid+profile+email
  &state=random_csrf_token
  &code_challenge=base64url(sha256(code_verifier))
  &code_challenge_method=S256
  3. 用户在 IdP 输入密码
    用户在 IdP 的登录页面输入账号密码（或扫码、生物识别等）。你的系统永远看不到这个密码。
  4. IdP 回调你的系统，携带 authorization code
    IdP 认证成功后，302 重定向回你在第2步指定的 redirect_uri，URL 中携带一个一次性的、短有效期的 authorization code。 
    GET /callback?code=xxxxx&state=random_csrf_token
  5. 后端用 code 换 Token
    你的后端服务器到服务器地向 IdP 的 /token 端点 POST，用 code + code_verifier 交换 ID Token 和 Access Token。 
    POST /token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=xxxxx
&redirect_uri=https://your-system.com/callback
&client_id=your_system_client_id
&client_secret=your_secret
&code_verifier=原始随机字符串
  6. 验证 ID Token，创建本地会话
    验证 ID Token 的签名、iss、aud、exp。提取 sub，查询是否已有绑定的本地账号。如有则直接登录，如无则走注册/绑定流程。创建自己系统的 Session/Cookie。

关键安全机制

PKCE（Proof Key for Code Exchange，发音 "pixy"）

即使 authorization code 被拦截，攻击者没有 code_verifier 也无法兑换 Token。 现代 OIDC 实现必须使用 PKCE，即使是后端应用。

state 参数

防止 CSRF 攻击。回调时验证 state 是否匹配，不匹配则拒绝。

nonce 参数

防止重放攻击。ID Token 中的 nonce 必须和请求时一致。

签名验证

使用 IdP 的公钥（JWKS）验证 ID Token 的签名，确保没有被篡改。

五、你的场景分析

📋 你的需求

✅ 你有一个自己的系统（系统A），用户可以在上面注册和登录

✅ 用户登录系统A后，访问另一个网站（系统B）希望自动登录

✅ 登录的是该用户在系统B中对应的账号（不是同一个账号，而是映射关系）

这种情况能不能用 SSO 实现？

答案：完全可以！这是 SSO 的经典场景。

你可以把你的系统A作为 IdP（身份提供商），系统B作为 RP（服务提供商）。当用户从系统A跳转到系统B时，系统B通过 OIDC 协议向系统A验证用户身份，实现自动登录。

具体怎么做到"对应另一个系统的账号"？

这是关键问题。系统B的用户账号和系统A的用户账号不是同一个数据库，需要建立映射关系。有几种方案：

方案原理优劣
1. 账号绑定（推荐） 用户首次通过 SSO 登录系统B时，系统B要求用户绑定已有账号或创建新账号。之后系统B存储 sub → 本地账号 的映射。 ✅ 最灵活
✅ 支持存量用户
⚠️ 首次需要绑定
2. ID Token 携带业务标识 系统A（IdP）在 ID Token 的 sub 或自定义 claim 中直接传递系统B的用户ID。需要系统A知道用户在系统B中的ID。 ⚠️ 系统A需要知道系统B的账号体系
⚠️ 耦合度高
3. 统一账号体系 系统A和系统B共用一套用户数据库。ID Token 中的 sub 就是统一用户ID。 ❌ 需要改造系统B
❌ 不适用于第三方系统
推荐方案1——账号绑定：流程和"用 Google 登录某某网站"一模一样。 首次登录时系统B问"检测到新账号，请绑定已有账号或注册"，之后就自动登录了。

六、落地：怎么把你的系统变成 IdP

你有两个选择

方案A：用现成的 IdP 服务

把你的用户导入到成熟的 IdP 平台，你的系统和系统B都对接它。

代表产品：
· Keycloak（开源，自部署）
· Auth0（SaaS）
· Zitadel（开源）
· Logto（开源）
· Azure AD / Entra ID（微软生态）

✅ 开发量小，功能全
✅ 安全合规（SOC2/GDPR）
⚠️ 需要迁移/同步用户

方案B：自己实现 OIDC IdP

在你的系统中直接实现 OIDC 协议，对外提供 /authorize、/token、/userinfo 等端点。

常用库：
· node-oidc-provider（Node.js）
· IdentityServer（.NET）
· oidc-provider（Python / Django）
· Casdoor（Go）
· Ory Hydra（Go，云原生）

⚠️ 开发量大，需要深入理解协议
⚠️ 安全责任在你（Token 签名、密钥轮换等）
✅ 无需迁移用户数据

方案A 架构示例（使用 Keycloak）

用户先登录系统A（系统A用 Keycloak 做认证），之后访问系统B时自动完成 SSO

方案B 架构示例（你的系统直接做 IdP）

系统B需要做什么

如果系统B是你也能控制的，它只需要：

  1. 引入 OIDC Client 库（任何语言都有现成的）
  2. 配置 IdP 信息：你的系统A的 issuer URL、client_id、client_secret
  3. 添加回调端点：处理 /callback 路由
  4. 建立账号映射表：idp_sub → local_user_id
  5. 改造登录入口：加一个"用系统A登录"按钮，或检测到系统A的 Session 后自动跳转

如果系统B是第三方系统（你改不了代码）？

这种情况比较麻烦：需要系统B本身就支持 SSO/OIDC 协议（企业 SaaS 常见的"配置 SSO"功能）。 如果系统B完全不支持，你无法通过标准 SSO 实现自动登录。 可能的替代方案包括：反向代理注入 Cookie、浏览器插件等，但这些都不是"干净"的方案。

# 你的系统A：OIDC IdP 端点实现概要

from fastapi import FastAPI, Request, HTTPException
from authlib.jose import jwt, JWTClaims
import time, secrets, hashlib

app = FastAPI()

# ===== 1. 发现端点（OIDC Discovery） =====
@app.get("/.well-known/openid-configuration")
async def openid_configuration():
    return {
        "issuer": "https://your-system-a.com",
        "authorization_endpoint": "https://your-system-a.com/authorize",
        "token_endpoint": "https://your-system-a.com/token",
        "userinfo_endpoint": "https://your-system-a.com/userinfo",
        "jwks_uri": "https://your-system-a.com/jwks",
        "scopes_supported": ["openid", "profile", "email"],
        "response_types_supported": ["code"],
        "id_token_signing_alg_values_supported": ["RS256"],
        "code_challenge_methods_supported": ["S256"],
    }

# ===== 2. 授权端点 =====
@app.get("/authorize")
async def authorize(request: Request):
    # 验证参数：client_id, redirect_uri, response_type, scope, state, code_challenge
    client_id = request.query_params["client_id"]
    redirect_uri = request.query_params["redirect_uri"]
    state = request.query_params.get("state")
    code_challenge = request.query_params["code_challenge"]

    # 检查用户是否已登录你的系统A
    session_user = get_current_user(request)  # 你的现有Session机制
    if not session_user:
        # 未登录，重定向到你的登录页
        return RedirectResponse("/login?next=" + quote(str(request.url)))

    # 用户已登录，生成一次性 authorization code
    code = secrets.token_urlsafe(32)
    store_code(code, {
        "user_id": session_user.id,
        "client_id": client_id,
        "redirect_uri": redirect_uri,
        "code_challenge": code_challenge,
        "expires_at": time.time() + 600,  // 10分钟有效
    })

    # 重定向回系统B
    return RedirectResponse(f"{redirect_uri}?code={code}&state={state}")

# ===== 3. Token 端点（服务器到服务器） =====
@app.post("/token")
async def token(request: Request):
    form = await request.form()
    code_data = verify_and_consume_code(
        form["code"],
        form["code_verifier"],   # PKCE 验证
        form["client_id"],
    )
    if not code_data:
        raise HTTPException(400, "invalid_grant")

    user = get_user_by_id(code_data["user_id"])

    # 生成 ID Token（JWT）
    id_token = jwt.encode(
        {"alg": "RS256"},
        {
            "iss": "https://your-system-a.com",
            "sub": str(user.id),            # 用户在系统A的唯一ID
            "aud": code_data["client_id"],
            "exp": int(time.time()) + 3600,
            "iat": int(time.time()),
            "email": user.email,          # 可选
            "name": user.name,             # 可选
        },
        PRIVATE_KEY,  # RSA 私钥
    )

    return {
        "access_token": generate_access_token(user),
        "id_token": id_token,
        "token_type": "Bearer",
        "expires_in": 3600,
    }

系统B（RP）实现

# 系统B：OIDC Relying Party 实现

from authlib.integrations.starlette_client import OAuth

oauth = OAuth()
oauth.register(
    name="system_a",
    server_metadata_url="https://your-system-a.com/.well-known/openid-configuration",
    client_id="system_b_client_id",
    client_secret="system_b_secret_xxx",
    client_kwargs={"code_challenge_method": "S256"},
)

@app.get("/login/oidc")
async def login_via_system_a(request: Request):
    redirect_uri = "https://system-b.com/callback"
    return await oauth.system_a.authorize_redirect(request, redirect_uri)

@app.get("/callback")
async def oidc_callback(request: Request):
    # 用 code 换 token
    token = await oauth.system_a.authorize_access_token(request)
    id_token = token["id_token"]       # 已自动验证签名

    sub = id_token["sub"]               # 用户在系统A的唯一ID
    email = id_token.get("email")

    # 查映射表
    binding = await db.query(
        "SELECT local_user_id FROM oidc_identity_binding"
        " WHERE idp_issuer = :iss AND idp_sub = :sub",
        {"iss": "https://your-system-a.com", "sub": sub}
    )

    if binding:
        # 已绑定 → 直接登录
        user = get_user_by_id(binding.local_user_id)
        set_session(request, user)
        return RedirectResponse("/dashboard")
    else:
        # 未绑定 → 导向绑定页面
        request.session["pending_oidc"] = {"sub": sub, "email": email}
        return RedirectResponse("/bind-account")

@app.post("/bind-account")
async def bind_account(request: Request):
    pending = request.session.pop("pending_oidc")
    # 验证用户在系统B的密码
    user = authenticate_local_user(request.form["username"], request.form["password"])
    if not user:
        raise HTTPException(401)

    # 写入映射
    await db.execute(
        "INSERT INTO oidc_identity_binding (idp_issuer, idp_sub, local_user_id) VALUES (:iss, :sub, :uid)",
        {"iss": "https://your-system-a.com", "sub": pending["sub"], "uid": user.id}
    )
    set_session(request, user)
    return RedirectResponse("/dashboard")

// 系统B：OIDC Relying Party（使用 openid-client）

const { Issuer, generators } = require('openid-client');

// 自动发现 IdP 配置
const systemAIssuer = await Issuer.discover(
  'https://your-system-a.com'
);

const client = new systemAIssuer.Client({
  client_id: 'system_b_client_id',
  client_secret: 'system_b_secret_xxx',
  redirect_uris: ['https://system-b.com/callback'],
  response_types: ['code'],
});

// 登录入口
app.get('/login/oidc', async (req, res) => {
  const codeVerifier = generators.codeVerifier();
  const codeChallenge = generators.codeChallenge(codeVerifier);
  req.session.codeVerifier = codeVerifier;

  const authUrl = client.authorizationUrl({
    scope: 'openid profile email',
    code_challenge: codeChallenge,
    code_challenge_method: 'S256',
  });
  res.redirect(authUrl);
});

// 回调处理
app.get('/callback', async (req, res) => {
  const params = client.callbackParams(req);
  const tokenSet = await client.callback(
    'https://system-b.com/callback', params,
    { code_verifier: req.session.codeVerifier }
  );

  const { sub, email } = tokenSet.claims();

  // 查映射 → 绑定或登录
  const binding = await db.query(
    'SELECT local_user_id FROM oidc_identity_binding WHERE idp_sub = :sub',
    { sub }
  );

  if (binding) {
    req.session.userId = binding.local_user_id;
    res.redirect('/dashboard');
  } else {
    req.session.pendingOidc = { sub, email };
    res.redirect('/bind-account');
  }
});

九、常见问题 FAQ

Q: SSO 和"免密登录"是一个东西吗？

不完全一样。"免密登录"更广，包括：密码管理器自动填充、Passkey/WebAuthn（指纹/Face ID）、Magic Link（邮箱链接登录）、短信验证码登录等。SSO 是免密登录的一种实现方式——用户第一次在 IdP 登录后，访问其他系统不需要再输入密码。

Q: 如果系统A和系统B在不同域名下，Session 能共享吗？

不能直接共享 Cookie。这正是需要 OIDC/SAML 协议的原因——通过基于重定向的协议来跨域名传递认证信息，而不是依赖 Cookie 共享。

Q: 用户在系统A退出登录了，系统B也会自动退出吗？

需要额外实现 Single Logout (SLO)。OIDC 有 RP-Initiated Logout 和 Back-Channel Logout 规范。简单方案：设置较短的 Session 过期时间 + 每次访问系统B时主动向 IdP 校验。

Q: 我的系统A的用户密码是 bcrypt 哈希的，能直接给 Keycloak 用吗？

可以。Keycloak 支持用户迁移——你可以写一个 Provider 插件，将认证委托给你的系统A API，或批量导入用户时直接保留 bcrypt 哈希（Keycloak 支持 bcrypt）。

Q: 系统B是别人的 SaaS 产品（比如 Jira、Slack），怎么对接？

这些企业 SaaS 通常有"配置 SAML/OIDC SSO"的功能（在管理后台的"安全"或"认证"设置中）。你提供 IdP 的 metadata URL，它们就能对接。如果系统B完全没有 SSO 功能，那就无法通过标准方式实现。
