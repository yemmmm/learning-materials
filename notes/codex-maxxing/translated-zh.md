# Codex-maxxing：让长时间运行的工作持续下去

**作者：** Jason Liu
**原文链接：** https://jxnl.co/writing/2026/05/10/codex-maxxing/
**OpenAI 白皮书：** https://openai.com/index/codex-maxxing-long-running-work/
**发布时间：** 2026 年 5 月 10 日

---

在 Codex 之前，我已经大量使用编程类 Agent。但大多数时候，我是通过为编程工作构建的界面来使用它们的：生成 diff、修改仓库、提交代码。

大约从 11 月开始，我把它们推向了知识工作领域。我用 Slidev 做演示文稿，把 Agent 当作带语音输入的记录员，并不断寻找编程 Agent 能帮我产出的其他产物：一个 `index.html`、一份 PDF、一个电子表格、一套幻灯片。

最新的 Codex 应用升级是我用过的第一个让这种更宽泛的工作模式感到"原生"的工具。Codex 在编程方面仍然很出色，但更有意思的转变是——它给了我的工作一个"栖身之处"。

真正改变我行为的，是学会了给工作建立一个**运行回路**：一条持久化的会话、共享记忆、能够操作我电脑的工具、能够引导和恢复任务的方式，以及一个让我审阅产物本身的界面。

## 持久化会话（Durable threads）

第一个改变我行为的是**压缩（Compaction）**。

> **Compaction（压缩）**
> 把一条长时间运行的会话压缩，使其能够继续下去，而不必完整保留每一条旧消息。

现在我给每一个我关心的重要工作流都保留了一条**置顶会话（pinned thread）**：

- 我的"参谋长"（Chief of Staff）会话
- Agents SDK
- OpenAI CLI
- Codex 开源项目
- 一条专门用来监控 Twitter 的会话

这些不是简短的聊天。它们是我已经压缩了好几个月的"超级会话（megathreads）"。它们累积了历史、偏好和旧的决策，而我不想每次回来都重新创建这些上下文。

> **置顶会话快捷键**
> 你可以用 `Command-1` 到 `Command-9` 直接跳转到置顶会话。

这里有一个权衡。长时间运行的会话不是免费的。如果你稍后再回来，会话内容很可能已经不在缓存中了，所以你会比在一条新的短会话里花费更多成本。但对于我真正在意的工作流，**连续性是值得的**。

## 语音输入（Voice input）

语音输入让 Codex 能够接收到我**更真实的思考**。

它的好处不是速度，而是 Agent 能够拿到我**未编辑过**的思考版本。Codex 自带语音输入，但我同时也用 Wispr Flow，因为系统级的听写改变了我能向其他所有东西输入的上下文量。如果我正在规划一项工作，我可能会说："我记得 Slack 里有个叫 Ben 的人提过这件事，我记不清具体是什么了，你自己去找一下。" 这种话打出来太模糊又烦人，但说出来却完全自然。

同样的道理也适用于**录音转写**。如果我想写一篇文章，我可以打电话给某人、录下对话，或者当面聊（用 Granola 在我手机上录音），然后用转写内容作为初始素材。当模型能够拿到我思考的"凌乱版本"而不仅是"打磨版本"时，很多计划都会变得更好。

## 引导（Steering）

语音和**引导**结合时会更有用。

> **Steering（引导）**
> 在 Codex 已经在工作时添加更多指令，而不是等待当前步骤完成。

引导允许你在一次工具调用之后插入下一条消息。如果我在审查一个网站，我可以一边看一边不停地说：

- 把这个缩小一点
- 这段文案是错的
- 这两个东西之间的间距感觉不太对
- 这个做完之后，开一个 PR
- 等预览部署完成
- 把预览链接发给需要在 Slack 上审查的那个人

我**不需要等每一步完成**才决定下一步。我可以一边让 Agent 继续工作一边不断添加意图，然后走开时整个任务队列已经被塑形好了。

稍后，**Heartbeats** 可以在我离开后监控那个 PR 或 Slack 会话。**工作的单位**不再是"一次提示一次回答"，而变成了**一个小的运行回路**。

## 记忆（Memory）

当会话开始持续更长时间后，它们就需要**在任何单一仓库之外的共享记忆**。

关键的举动是**把东西存到磁盘上**。一条长会话可以记住很多东西，但那种记忆是被困在会话内部的，除非有用的部分被写到某个持久化的地方。记忆系统的意义，就是把会话学到的东西转化成**我可以检查、编辑、对比和复用的产物**。

我大多数长时间运行的会话都从一个 Obsidian 知识库（vault）开始：

```
vault/
├── TODO.md
├── people/
├── projects/
├── agent/
└── notes/
```

在最顶层，我保留了一个 `AGENTS.md` 指令文件，告诉模型**把事情记下来**：当你对人了解得更多、在项目上取得进展、做出决策或闭合一个未完成的事项时，更新知识库里相关的页面。

**这个知识库就是 Agent 的住所，独立于任何一个项目。** 仓库（repo）存放代码。知识库存放围绕我工作的**流动上下文**：人、决策、未完成的事项、每日笔记、项目状态，以及那些否则会在会话之间丢失的理解片段。

我还把这个知识库作为 GitHub 仓库保留。这给我带来两样东西：

1. 它可以在云端工作
2. diff 成了记忆的**审查界面**

当 Agent 更新知识库时，我可以阅读 diff，看看它认为哪些东西重要到值得记住。**这一步审查很关键。** 我不希望那些常驻会话在对话历史里悄悄累积"氛围感（vibes）"。我希望它们**把变化写下来**：这个人偏好这个、这个项目在等那个、做出了这个决策、这个事项已经闭合。

这也是我喜欢"以文件形式承载记忆"的原因。文件迫使 Agent 把经验压缩成一种**能在会话消亡后存活的形态**。如果会话挂了、压缩坏了，或者依赖它变得太昂贵，有用的知识仍然在那里。模式很简单：**把重要的工作保存在一个仓库里，给 Codex 权限和指令去更新它，然后像审查任何其他改动一样审查它的 diff。**

到了这个阶段，置顶会话开始感觉不再像是聊天，而更像是**不同的工人从同一本笔记本里读东西**。

Codex 在 `Settings > Personalization > Memories` 里也有第一方记忆功能。我把它看作是显式磁盘记忆系统之上的一层**召回层**。**知识库是我可以审查和编辑的事实来源（source of truth）。** Memories 则帮助 Codex 在我开始一条新会话时记住稳定的偏好、重复出现的工作流、项目约定和已知的坑。

**Chronicle** 出于同样的原因也很有趣。我还没认真用过它，文档也明确指出它是一个**研究预览版（opt-in）**，在权限、速率限制、提示注入以及未加密的本地记忆文件方面有真实的权衡。但从方向上看，它指向了我在意的同一件事：**工作应该留下结构化的记忆，而不只是一段更长的聊天记录。**

> **Shared memory（共享记忆）**
> 保存在单一聊天之外的上下文，例如我知识库里的笔记，不同的会话都可以复用。

## 电脑与浏览器使用（Computer and Browser Use）

一旦会话有了记忆，下一个问题就是：**它能触达什么？**

我头脑中最有用的区分是：

- `$browser` 用于我想要**检查和标注的本地 Web 界面**
- `@chrome` 用于**已登录的浏览器状态和多个标签页**
- `@computer` 用于**只以 GUI 形式存在的工作**

如果我在迭代一个本地应用，我用 `$browser`。如果我需要在已登录的浏览器会话里工作，我用 `@chrome`。如果完成任务的唯一方式是点击穿过某个桌面应用，我用 `@computer`。

**Appshots** 是这件事的轻量级版本。有时候我还不想让 Codex 操作这个应用，我只想**给它看我现在正在看的东西**。在 macOS 上，你可以同时按两个 Command 键，把最前方的窗口连同截图和应用能提供的任何文本，发送到一条 Codex 会话里。

这对于那种**"展示比描述更容易"的中间地带上下文**非常有用：一个错误弹窗、一个设置面板、一个 API 参考页面、一封邮件、一个日历视图、一个设计预览，或者某个应用只有在看到时才说得通的奇怪状态。我不再需要打一长段设置性提示词，我可以指向当前窗口说："这就是我说的那个东西。"

我的工作机上，Twitter 登录在 Safari 里。如果我让 `@computer` 在那里读 Twitter，它在工作时就会占用 Safari。当我想让 Agent **并行使用多个已认证的标签页，而不接管我正在使用的整个应用**时，`@chrome` 更好。

**Connectors（连接器）** 把这种触达延伸到了我实际工作的其他地方。我用得最多的是 `$slack`、`$gmail` 和 `$calendar`，因为 Slack 会话、收件箱和日历是许多工作在变成代码**之前**最先出现的地方。

**Skills（技能）** 让重复的工作流变得可复用。Skill Creator 和 Skill Installer 是个不错的起点。Skill Installer 让你直接从输入框添加 OpenAI 推荐的技能。在 Codex 宠物（Codex pets）发布后，我用它安装了 Hatch Pet 技能，但这个有用的模式是通用的：**一旦你做过一次有用的事情，你通常可以把它打包起来，让 Codex 不用重新教就能再做一次。**

## 远程控制（Remote control）

远程控制让这些更长的回路变得**可携带**。

Codex 可以在你那台文件、权限和本地环境都已经存在的机器上继续工作，同时你从手机端查看它发现了什么、回答一个问题、批准下一步，或者改变方向——**不必回到桌前**。OpenAI 把它描述为"从任何地方与 Codex 协作"的方式。

这在 Codex 已经在做某个长时间运行的任务、而你想**保持势能**时最有价值。你可以启动一个任务、走开，然后在它到达决策点时用手机引导它。

这件事之所以重要，和置顶会话、语音、Heartbeats 重要的原因是**同一个**：工作不再因为我换了地点而被迫暂停。会话可以继续走，而我只需要保持刚好足够的注意力去**解开下一步**。

## Heartbeats（心跳/定时巡检）

置顶会话很有用，但它们仍然在**等你说话**。**Heartbeats 让它们能够自我循环。**

> **Heartbeats**
> 一条会话可以为自己安排的**定期检查**，比如盯着 Slack 或某个 PR 等待新动态。

一个 Heartbeat 是**会话本地的自动化**。你可以说"每隔几个小时关注一下这件事"，这条会话就能为自己排定时间。一条会话可以有多个时间表，可以一直运行到某个条件被满足，并随着时间调整自己的节奏。

### 参谋长（Chief of Staff）

我的参谋长会话每 30 分钟运行一次：

```
每 30 分钟检查一次 Slack 和 Gmail，找出需要我关注的未回消息。
帮我确定最重要的事情的优先级。
如果有人问我问题，尽可能深入地研究答案，并为我起草一份回复，但不要发送。
```

当我回到 Slack 时，回复往往已经躺在草稿里了。**仍然由我决定什么被发出去**，但**收集上下文这个昂贵的部分已经做完了**。

### 监控反馈

同样的模式也适用于审查回路。一个 Heartbeat 可以监控 Google Docs 评论、PR 评论或 Slack 回复，并在反馈到来时让工作继续推进。

我最喜欢的一个例子来自一个动画项目。我在 Slack 里发了一个视频，并让 Codex 每 15 分钟检查一次会话看有没有反馈，当评论出现时重新渲染一个新版本，然后回到会话里 @ 那个审查者。Slack 的 MCP 服务器不能上传文件，所以 Agent 就用 `@computer` 按下"添加文件"按钮，把改好的渲染版本发了出去。

有意思的部分不只是它每 15 分钟检查一次 Slack。**这个回路跨越了工具边界**：Slack 负责反馈，Remotion 负责渲染，`@computer` 负责上传。也就是在这个时刻，Heartbeats、连接器和电脑使用不再像是各自独立的功能——它们合在一起，变成了**一个不需要我守在原地、能持续运行的反馈回路**。

### 拿到退款

最近我有一个包裹被偷了。Amazon 告诉我大概需要 25 分钟才能等到人工客服，所以我建了一条带 `@computer` 的会话，告诉它：

```
每 5 分钟检查一次客服是否加入了这个会话。
如果加入了，尽最大努力帮我拿到退款。
一旦他们回复，切换到每分钟检查一次，以便你能更快响应。
```

等我洗完澡出来，**退款已经办好了**。

我的很多 Heartbeats 也会更新我的 Obsidian 知识库，作为一种显式记忆。

## 目标（Goals）

最新的、我还在学习如何用好用的东西是 **Goals**。

> **Goals（目标）**
> 有真正终点线的、较长时间运行的任务，Codex 可以持续朝它推进。

**你应该对它们保持雄心。** 一个弱目标是"按照这个 Markdown 文件里的计划去实现"。一个强目标有**真正的成功标准**，让 Agent 能够不断朝它推进。

上周我尝试把 Python 的 Rich 库迁移到 Rust。因为原项目已经有一套庞大的单元测试，我可以设定这样的目标：把 Rich 迁移到 Rust，**但它必须通过原库的所有单元测试**。

那套测试给了这次运行一个**真正的预言机（oracle）**：Rust 移植在通过和 Python 库相同的测试之前，**不算完成**。

这和"跟 AI 进行一段长对话、累积一份 Markdown 计划、然后最终说'实现它'"**不一样**。**执行的好坏，取决于你给它的目标和验证方式。没有验证的雄心，只是一个愿望。**

## 侧边栏（The side panel）

Codex 让我**最兴奋**的部分是侧边栏。

很容易把它想成"做预览的地方"。这低估了它。**侧边栏是 Codex 不再只是一个聊天应用、而开始变成"工作发生的地方"的所在。**

对我来说它做三件事：**检查产物、操作 Web 界面、审查改动**。在这三件事中，我都可以看着 Agent 正在操作的同一个对象，并对它做出评论。

### 检查产物

Markdown、电子表格、CSV、PDF 和幻灯片都可以放在这里。

Markdown 可以评论。电子表格会渲染公式并支持单元格编辑，我用它来管理 Codex 开源计划。CSV 会变成表格而不是原始文本。PDF 直接渲染，这对 LaTeX 特别有用。幻灯片可以不用离开应用就创建和审查。

重要的事情**不只是 Codex 能生成产物**，而是**我可以检查和标注它们而不打断回路**。

### 操作 Web 界面

应用内的浏览器更有意思。Agent 能看到它，通过 `$browser` 用 JavaScript 控制它，而我可以**直接在我正在看的东西上留下标注**。

有几个 Web 界面我现在一直在这样用：

- `index.html` 用于轻量级静态产物
- Storybook 用于审查 UI 组件
- Remotion Studio 用于程序化动画
- Slidev 用于演示文稿
- Streamlit 用于数据应用

**最小版本往往就是最好的。** 你可以让模型做一个单一的 `index.html` 文件（带 JavaScript 和 CSS），在侧边栏打开它，立刻开始交互——**不需要服务器**。我一直在实验用 Heartbeats 随时间更新一个静态 `index.html`，这样每当我回到一条会话时，**那里已经有一个新的产物在等我了**。

Thariq 有一篇很好的文章，讲为什么倾向于用 HTML 而不是 Markdown 作为输出格式。我觉得这个直觉是对的。**一旦输出是一个小应用而不是一份文档，关系就改变了。**

如果我需要更重的东西，我可以用 Vite 应用，但那样我就得保持一个服务器在运行。一个普通的 `index.html` 要持久得多。

做动画工作时，我经常把 Storybook 和 Remotion Studio 并排打开。我可以留下"让这个弹一下"或"这个应该更大一点"这样的评论，而 Agent 可以检查**我正在看的同一个浏览器状态**，包括动画里当前的那一帧。

做演示文稿时，我经常用 Slidev。Codex 可以检查幻灯片、抓到被截断的内容、在幻灯片之间切换，并在我审查时回应我的标注。

我也预期这会对 Streamlit 和 Jupyter 这类工具越来越有用。不同的人本来就生活在不同的应用里。**Codex 可以越来越多地到他们所在的地方去找他们。**

Codex 拥有的**可以记忆、可以重访、可以检查、可以行动**的地方越多，我的工作在两次提示之间**死掉**的就越少。这就是我在意的改变——**不是 Agent 能替我写代码，而是更多的工作可以在你离开之后继续向前走。**
